# 灵感库 + 工作流跟踪 Skill

## 核心架构

本 Skill 采用**两层架构**：

### 层 1：灵感库（知识存储）
自动捕获用户在对话中提到的：
- 论文（Paper）/ 链接（URL）/ 灵感（Idea）
- 自动分类 + 打标签
- 超过阈值（默认 50 条）自动压缩成综述文档

### 层 2：工作流状态跟踪（思路保持）
在 AI 长工作流中保持清晰：
- 记录了什么（What）
- 进度怎么样（Progress）
- 遇到了什么困难（Blockers）
- AI 推荐下一步干什么（Next Steps）

---

## 触发词

以下场景自动触发（无需用户手动调用）：

| 触发场景 | 触发词示例 | 动作 |
|---|---|---|
| 灵感捕获 | 「我想到…」、「有个想法…」、「记一下…」 | 自动询问是否记录到灵感库 |
| 链接/论文分享 | 包含 `http(s)://` 或「论文」、「paper」 | 自动提取并询问是否存储 |
| 工作流更新 | 「我做了…」、「刚才…」、「现在在…」 | 更新工作流状态 |
| 遇到困难 | 「卡住了」、「不知道怎么办」、「报错」 | 记录困难 + 推荐下一步 |
| 查询灵感 | 「之前想到的…」、「我记录过…」 | 检索灵感库 |
| 查看进度 | 「现在进度如何」、「做到哪了」 | 输出工作流状态总结 |
| 导出总结 | 「导出记忆」、「生成总结」、「写一份报告」 | 导出格式化的记忆/工作流报告 |

## 工作流

### Step 1：灵感捕获

当用户提到灵感/论文/链接时：

1. **识别**：AI 通过语义识别用户是否在分享灵感、论文或链接
2. **确认**：询问用户是否要记录（避免误捕获）
3. **提取**：
   - 标题（自动生成或让用户确认）
   - 内容（完整记录）
   - 类型（`paper` / `url` / `idea` / `decision`）
   - 标签（自动提取关键词 + 用户确认）
   - 关联灵感（AI 自动检索库中相关条目）
4. **存储**：写入灵感库（见「存储方案」）

**示例对话**：
```
用户：「刚才看到一篇关于 RAG 优化的论文，感觉可以用在我们的知识库里」

AI：「✅ 已识别到灵感：
    标题：RAG 优化论文 → 可应用于知识库
    类型：paper
    标签：#RAG #知识库 #优化
    
    要记录到灵感库吗？（Y/n）」
```

---

### Step 2：自动分类与标签

每条灵感存储时自动：
1. **分类**：`paper` / `url` / `idea` / `decision` / `blocker` / `milestone`
2. **打标签**：
   - 自动提取关键词（AI 生成）
   - 用户可修改
   - 支持多级标签（`#AI/提效` `#工程/后端`）
3. **关联**：
   - AI 自动检索库中相似灵感
   - 询问用户是否建立关联
   - 关联后可在检索时一并展示

---

### Step 3：阈值检测与自动压缩

**触发条件**：
- 灵感库条目数 ≥ **50 条**（可配置）
- 或 单个标签下条目数 ≥ **20 条**（可配置）

**压缩动作**：
1. AI 读取所有灵感
2. 按标签/类型分组
3. 生成**综述文档**（Markdown 格式）：
   - 每个标签生成一段总结
   - 提取核心洞察
   - 标注灵感之间的关联
4. 存为**综述文档**（`reviews/review-YYYY-MM-DD.md`）
5. 可选：将已压缩的灵感标记为 `archived`（不从库里删除，只是标记）

**示例综述文档结构**：
```markdown
# 灵感综述 - 2026-06-09

## #RAG（12 条灵感）
核心洞察：RAG 优化的方向集中在检索层（混合检索）+ 重排层（Cross-Encoder）
关联灵感：#向量数据库 #知识库

### 关键灵感
1. [06-05] RAG 优化论文 - 提出混合检索方案
2. [06-07] 我们的知识库可以用 RAG - 需要确认数据格式
...

## #AI/提效（8 条灵感）
...
```

---

### Step 4：工作流状态跟踪

**记录内容**：
| 字段 | 说明 | 示例 |
|---|---|---|
| `timestamp` | 时间戳 | 2026-06-09 12:30 |
| `action` | 做了什么 | 完成了灵感库 SKILL.md 编写 |
| `progress` | 进度（0-100%） | 30% |
| `blockers` | 遇到的困难 | tai-skill 鉴权脚本有 bug |
| `next_steps` | AI 推荐下一步 | 修复 token 缓存逻辑 |

**触发时机**：
- 用户说「我做了…」→ 记录 `action`
- 用户说「完成了…」→ 更新 `progress`
- 用户说「卡住了…」→ 记录 `blocker` + 推荐 `next_steps`
- 用户说「接下来干嘛」→ 输出当前状态 + 推荐 `next_steps`

**状态存储**：`.workbuddy/memory/workflow-state.json`

---

### Step 5：检索与输出

**检索方式**：
1. **语义检索**（默认）：输入自然语言，返回最相关的 N 条灵感
2. **标签过滤**：`查看 #RAG 的灵感`
3. **时间范围**：`查看上周的灵感`
4. **类型过滤**：`查看所有 paper` / `查看所有 idea`

**输出格式**：
```
找到 3 条相关灵感：
1. [06-05] RAG 优化论文（关联度 94%）
   标签：#RAG #知识库 #优化
   关联：[06-07] 知识库应用 RAG
   
2. [06-07] 知识库应用 RAG（关联度 87%）
   ...
```

---

### Step 6：导出

**导出格式**：
1. **Markdown**（默认）：适合 KM / 腾讯学堂发文
2. **JSON**：适合程序化处理
3. **Mermaid 知识图谱**：可视化灵感关联
4. **工作流报告**：包含「做了什么 / 进度 / 困难 / 下一步」

**导出触发**：
- 用户说「导出灵感」→ 导出灵感库
- 用户说「导出工作流」→ 导出工作流状态
- 用户说「导出总结」→ 导出两者 + 综述

**导出位置**：`.workbuddy/memory/exports/`

## AI 行为规则

### ✅ 应该做的
1. **主动识别**：在对话中主动识别灵感/论文/链接，不要等用户说「记录」
2. **确认后再存**：识别到后先询问，避免误存
3. **自动关联**：存储时自动检索相关灵感并询问是否关联
4. **定期提醒**：每 10 条灵感提醒用户「要不要看看现在的灵感分布？」
5. **压缩前确认**：达到阈值后先询问用户，不要自动压缩

### ❌ 不应该做的
1. **不要强制**：用户说「不记录」就跳过，不要说服
2. **不要过度捕获**：日常聊天不要当成灵感（除非用户明确说「记一下」）
3. **不要频繁打断**：识别到灵感后，等用户说完一段话再询问
4. **不要删除原始数据**：压缩后原始灵感保留，只是标记 `archived`

```

├── inspirations.json        # 灵感库（主文件）
├── workflow-state.json      # 工作流状态
```

## 配置项

用户可通过对话修改配置：

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `compression_threshold` | 50 | 灵感数达到此值触发压缩 |
| `auto_detect` | true | 是否自动识别灵感 |
| `confirm_before_save` | true | 存储前是否询问确认 |
| `reminder_interval` | 10 | 每 N 条灵感提醒一次 |
| `compression_mode` | "confirm" | 压缩模式：`auto` 自动 / `confirm` 确认后 / `manual` 手动 |

**修改示例**：
```
用户：「把压缩阈值改成 30 条」
AI：「✅ 已修改压缩阈值为 30 条」
```

## 与现有记忆系统的关系

本 Skill 是 `.workbuddy/memory/` 的**上层封装**：

| 现有机制 | 本 Skill 的角色 |
|---|---|
| `YYYY-MM-DD.md`（日记） | 自动从中提取灵感（可选） |
| `MEMORY.md`（长期记忆） | 将压缩后的综述同步到 MEMORY.md |
| `workflow-state.json`（新增） | 专门跟踪工作流状态 |

**协同工作**：
- 用户说「记录灵感」→ 写入 `inspirations.json` + 可选同步到 `YYYY-MM-DD.md`
- 压缩后 → 综述文档 + 同步核心洞察到 `MEMORY.md`
- 工作流更新 → 写入 `workflow-state.json` + 同步到 `YYYY-MM-DD.md`

## 技术实现说明

本 Skill 的实现依赖：
1. **文件 I/O**：读写 `.workbuddy/memory/inspirations/` 下的 JSON/MD 文件
2. **语义检索**：可用 `grep` 做关键词搜索，未来可接入向量数据库（如 Chroma）
3. **自动压缩**：AI 读取 `inspirations.json` → 生成综述 → 写入 `reviews/`
4. **工作流跟踪**：AI 维护 `workflow-state.json` 的状态

**MVP 阶段**（本次实现）：
- 使用 JSON 文件存储（不依赖数据库）
- 关键词搜索（不依赖向量数据库）
- AI 手动触发压缩（不自动）

**未来优化**：
- 接入向量数据库（Chroma / FAISS）
- 支持 Device Code 流鉴权（无头环境）
- 多用户隔离（团队协作）
